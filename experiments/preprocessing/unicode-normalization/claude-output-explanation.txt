  The Core Problem: One Sound, Multiple Computer Representations

  What's Actually Happening in Your Data

  When you type the Aja word with a tone-marked vowel like ɔ́ (open-o with high tone), your computer can store it in two 
  completely different ways - but they look identical to your eyes!

  Method 1: Decomposed (NFD)
  ɔ́  =  ɔ  +  ´
      (base) (combining accent)
      2 separate characters
      U+0254 + U+0301

  Method 2: Composed (NFC)
  ɔ́  =  ɔ́
      1 single character
      U+1ECD (or precomposed if exists)

  Both look exactly the same on screen: ɔ́

  But to a computer (and ML models), they are completely different.

  "Tom can Marie can, wowo dele bɔbɔ lɔ mɛ ò."

  - ò (single precomposed character)
  - = 1 byte in the file

  Why This Breaks Machine Learning

  Let me show you what happens when you train an ML translation model on inconsistent data:

  Scenario: Training Without Normalization

  Your dataset has the word "drɔ" (to know/understand) appearing 50 times:
  - 30 times: drɔ (composed: d + r + ɔ as single char)
  - 20 times: drɔ (decomposed: d + r + ɔ + combining accent as 2 chars)

  What the model sees:
  Token 1: "drɔ"     (appears 30 times)
  Token 2: "drɔ"     (appears 20 times)  ← Model thinks this is different!

  The Problem:
  1. Model learns two separate vocabulary entries for the same word
  2. Each entry only sees partial examples (30 vs 20 instead of 50 total)
  3. Model has less statistical evidence that "drɔ" translates to "know"
  4. Prediction quality drops

  After Normalization (NFC)

  Now ALL 50 instances use the same representation:
  Token: "drɔ"     (appears 50 times)

  The Benefit:
  1. Model learns ONE vocabulary entry
  2. Full statistical evidence (50 examples)
  3. Better translation quality
  4. Smaller vocabulary = more efficient model

  When you open your CSV in Excel, TextEdit, or any editor, both versions look identical:

  Decomposed: [0xC9, 0x94, 0xCC, 0x81] (2 codepoints)
  Composed: [0xC9, 0x94] (1 codepoint)
