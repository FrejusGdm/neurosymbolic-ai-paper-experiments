## 1. Prerequisites

### Local machine
- SSH access to Discovery: `ssh f006g5b@discovery.dartmouth.edu`
- `scp` for file transfers
- Data splits prepared locally (see `experiments/data/prepare_splits.py`)

### HPC environment
- HuggingFace token: `export HF_TOKEN="hf_xxxxx"` (add to `~/.bashrc`)
- Apptainer (Singularity successor) — pre-installed on Discovery
- SLURM job scheduler — pre-installed on Discovery

### Files needed in the HPC working directory
```
adja-nmt-hpc/
├── build_container.sh       # Step 1: builds container
├── nmt-training.def         # Apptainer recipe
├── download_models.sh       # Step 2: downloads model weights
├── generate_jobs_tsv.py     # Step 3: generates job parameter file
├── hf_job_train_hpc.py      # Training script (runs inside container)
├── submit_array.sbatch      # Step 4: SLURM submission script
├── upload_results.sh        # Step 5: push results to HF Hub
├── data/                    # Training data (uploaded via scp)
│   ├── shared/test.tsv
│   ├── exp1/{condition}/train.tsv, val.tsv
│   ├── exp2/{condition}/train.tsv, val.tsv
│   ├── baselines/{condition}/train.tsv, val.tsv
│   └── ablations/{type}/{condition}/train.tsv, val.tsv
├── models/                  # Downloaded by download_models.sh
│   ├── nllb-200-1.3B/      (~5 GB)
│   └── mbart-large-50/     (~2.4 GB)
├── jobs.tsv                 # Generated by generate_jobs_tsv.py
├── logs/                    # Created before submission, filled by SLURM
└── results/                 # Created by training jobs
```

## 2. Cluster Overview

### Checking available partitions
```bash
sinfo -s
```

### Checking GPU resources on a specific partition
```bash
# Basic view
sinfo -p gpuq -O gres,nodelist | head -20

# Full GRES string (not truncated)
sinfo -p gpuq -O gres:60,nodelist | head -20
```

Example output (Feb 2026):
```
GRES                                                        NODELIST
gpu:l40s:8(S:0-1)                                           l40sx8
gpu:nvidia_a100_80gb_pcie_3g.40gb:4(S:0-1)                  a01
```

### Checking your account/partition access
```bash
sacctmgr show assoc where user=$USER format=account,partition,qos
```

If you only see `free` account with blank partition, you may still have access
to `gpuq` — test with `sbatch --test-only` before submitting real jobs.

### Key partitions on Discovery
| Partition | GPUs | Typical Use |
|-----------|------|-------------|
| `gpuq` | A100 80GB, L40s | General GPU jobs |
| `v100` | V100 | Older GPU jobs |
| `2080ti` | 2080Ti | Small experiments |
| `preempt_*` | Various | Preemptible (can be killed) |

## 4. Storage Management

### Check disk usage
```bash
# Your home directory quota
df -h .

# What's using space
du -sh ~/* | sort -rh | head -20

# Specific directory size
du -sh ~/adja-nmt-hpc/models/*
```

### Home directory is limited (50GB)
Discovery home dirs are 50GB. Model weights + containers easily exceed this.

### Using lab storage (recommended for large experiments)
If your PI has lab storage, use it:
```bash
# Example: Professor Coto's lab storage (1000GB)
ls /dartfs/rc/lab/R/RCoto/godeme

# Copy your working directory there
cp -r ~/adja-nmt-hpc /dartfs/rc/lab/R/RCoto/godeme/adja-nmt-hpc

# Work from the new location
cd /dartfs/rc/lab/R/RCoto/godeme/adja-nmt-hpc
```

### Using scratch space (alternative)
```bash
# Check if scratch exists
ls /dartfs-hpc/scratch/$USER

# Move large files there
mv ~/adja-nmt-hpc /dartfs-hpc/scratch/$USER/adja-nmt-hpc
ln -s /dartfs-hpc/scratch/$USER/adja-nmt-hpc ~/adja-nmt-hpc
```

> **Warning**: Scratch space may be purged periodically. Lab storage is persistent.

## 5. Building the Container

### What it does
`build_container.sh` creates an Apptainer (Singularity) container with all Python
dependencies pre-installed. This ensures reproducible environments across compute nodes.

### Container recipe (`nmt-training.def`)
- Base: `pytorch/pytorch:2.1.0-cuda12.1-cudnn8-runtime`
- Added packages: `transformers==4.44.2`, `sentencepiece`, `sacrebleu`, `huggingface-hub`, etc.
- Sets `PYTHONUNBUFFERED=1` so print output appears in logs immediately

### Build command
```bash
cd /path/to/adja-nmt-hpc
bash build_container.sh
```

### What to expect
1. First run pulls the base PyTorch image (~5GB) — can take 10-20 minutes
2. Has retry logic (5 attempts, 30s between retries) for flaky network
3. Second step builds final container from local base — fast, no network needed
4. Produces `pytorch-base.sif` (~5GB) and `nmt-training.sif` (~6GB)

### How to know it succeeded
- You see: `Done: X.XG — nmt-training.sif`
- Files exist: `ls -lh *.sif`

### How to know it failed
- Script exits early with an error (uses `set -euo pipefail`)
- Common: network timeout during pull → retries automatically
- If `--fakeroot` fails, check `apptainer --version` and cluster docs

### Skips if already built
Both `.sif` files are checked before pulling/building. Safe to re-run.

## 6. Downloading Models

### What it does
`download_models.sh` downloads pretrained model weights into `models/` using
`huggingface_hub.snapshot_download()` inside the container.

### Models downloaded
| Model | Directory | Size | Source |
|-------|-----------|------|--------|
| NLLB-200-1.3B | `models/nllb-200-1.3B/` | ~5 GB | `facebook/nllb-200-1.3B` |
| mBART-large-50 | `models/mbart-large-50/` | ~2.4 GB | `facebook/mbart-large-50-many-to-many-mmt` |

### Download command
```bash
bash download_models.sh
```

### What to expect
- Downloads run inside the container via `apptainer exec`
- `HF_TOKEN` environment variable must be set
- Each model shows `"Done."` when complete
- Final message: `"All models downloaded to .../models/"`
- Followed by `ls -lh` of the models directory

### Skips if already downloaded
Checks for `config.json` in each model directory. Safe to re-run.

### Requires ~7.4GB free disk space
This is why lab storage or scratch is important — home directory may be too small.

## 7. Preparing Data

### Data is prepared locally, then uploaded

The data splits are generated on your local machine using `experiments/data/prepare_splits.py`.
The HPC scripts expect the data to already be in place.

### Upload from local Mac to HPC
```bash
# From your LOCAL terminal (not on HPC)
scp -r /path/to/experiments/data/splits \
  f006g5b@discovery.dartmouth.edu:/path/to/adja-nmt-hpc/data
```

The `splits/` directory becomes `data/` on the HPC. Only ~34MB — transfers in seconds.

### Expected data structure
```
data/
├── shared/
│   └── test.tsv              # 1,455 lines, shared across ALL experiments
├── exp1/
│   ├── RANDOM-10K/
│   │   ├── train.tsv
│   │   └── val.tsv
│   ├── STRUCTURED-4K-ONLY/
│   │   ├── train.tsv
│   │   └── val.tsv
│   └── ... (6 conditions total)
├── exp2/
│   ├── STRUCTURED-200/
│   ├── RANDOM-10000/
│   ├── RANDOM-6K_STRUCTURED-500/
│   ├── REPLACE-R9500_S500/
│   └── ... (30+ conditions)
├── baselines/
│   ├── LENGTH-STRATIFIED/
│   ├── VOCAB-MAXIMIZED/
│   └── TF-IDF-DIVERSE/
└── ablations/
    ├── module_loo/       (FULL, NO-NEGATION, NO-PAST, etc.)
    ├── module_size_ctrl/ (FULL-1K, NO-NEG-1K, etc.)
    ├── pronoun/          (ALL-8, REDUCED-4, etc.)
    ├── verb/             (10-VERBS, 5-VERBS-a, etc.)
    └── minimal_pairs/    (PAIRS-INTACT, PAIRS-BROKEN)
```

### TSV format
Tab-separated, no header: `French sentence\tAdja translation`
```

vous mangez du riz mi	ɖuɖuɔ mɔlu
```

### Verify data is in place
```bash
ls data/shared/test.tsv
ls data/exp1/ data/baselines/ data/exp2/ data/ablations/
wc -l data/shared/test.tsv   # should be 1455
```

## 8. Generating the Job File

### What it does
`generate_jobs_tsv.py` creates `jobs.tsv` — a parameter file where line N defines
the parameters for SLURM array task N.

### Run it
```bash
python3 generate_jobs_tsv.py
```

### Output
```
Generated jobs.tsv with 699 jobs
  NLLB-1.3B:     lines 1-233   (40 already done → use --array=41-699)
  mBART-fr:      lines 234-466
  mBART-rand:    lines 467-699
  Total:         699 jobs
```

### Job file format (TSV, no header)
```
job_id  experiment  condition  seed  model_key  results_subdir  similar_lang
1       exp1        RANDOM-10K 42    nllb-1.3b  nllb-1.3b       ewe_Latn
2       exp1        RANDOM-10K 123   nllb-1.3b  nllb-1.3b       ewe_Latn
...
234     exp1        RANDOM-10K 42    mbart-50   mbart-fr         fr_XX
...
467     exp1        RANDOM-10K 42    mbart-50   mbart-rand       none
```

### 233 conditions × 3 model configs = 699 jobs
| Model Config | Lines | results_subdir | similar_lang | What it does |
|-------------|-------|----------------|--------------|--------------|
| NLLB-1.3B | 1-233 | `nllb-1.3b` | `ewe_Latn` | Transfer from Ewe (related Gbe language) |
| mBART French-init | 234-466 | `mbart-fr` | `fr_XX` | Transfer from French embeddings |
| mBART Random-init | 467-699 | `mbart-rand` | `none` | No transfer (control) |

## 9. Submitting Training Jobs

### Pre-submission checklist
```bash
# 1. Are you in the right directory?
pwd   # should be /path/to/adja-nmt-hpc

# 2. Container exists?
ls -lh nmt-training.sif

# 3. Models downloaded?
ls models/nllb-200-1.3B/config.json models/mbart-large-50/config.json

# 4. Data in place?
ls data/shared/test.tsv

# 5. Jobs file generated?
wc -l jobs.tsv   # should be 699

# 6. Logs directory exists?
mkdir -p logs

# 7. HF_TOKEN set? (needed for upload later)
echo $HF_TOKEN
```

### Dry run (check permissions without actually submitting)
```bash
sbatch --test-only --array=1-1 submit_array.sbatch
```
If this succeeds, you have access to the partition.

### Submit all jobs
```bash
# Fresh start (all 699 jobs):
sbatch --array=1-699 submit_array.sbatch

# If NLLB jobs 1-40 already ran on HF Jobs:
sbatch --array=41-699 submit_array.sbatch

# Submit by model (useful for debugging):
sbatch --array=1-233 submit_array.sbatch     # NLLB only
sbatch --array=234-466 submit_array.sbatch   # mBART French-init only
sbatch --array=467-699 submit_array.sbatch   # mBART Random-init only

# Submit a single job for testing:
sbatch --array=1-1 submit_array.sbatch
```

### SBATCH resource requests
```
Partition:  gpuq
GPU:        nvidia_a100_80gb_pcie_3g.40gb:1  (1 MIG slice of A100)
CPUs:       4
Memory:     32 GB
Time limit: 4 hours per job
```

### Auto-skip existing results
The submission script checks for `results/{subdir}/{exp}/{cond}/seed{seed}/test_metrics.json`.
If it exists, the job exits immediately with `"SKIP: Results already exist"`.
This makes it **safe to re-submit the full range** — completed jobs finish instantly.

### IMPORTANT: Always `cd` into the directory before submitting
SLURM writes log files (`logs/nmt-%A_%a.out`) relative to your **current working directory**,
not the script location. If you submit from a different directory, logs end up in the wrong place.

## 10. Monitoring Jobs

### Check job queue
```bash
# All your jobs
squeue -u $USER

# Just running jobs
squeue -u $USER -t RUNNING

# Just pending jobs
squeue -u $USER -t PENDING

# Compact summary
squeue -u $USER -o "%.8i %.9P %.8j %.2t %.10M %.6D %R" | head -30
```

### Check job accounting (after jobs finish)
```bash
# Summary of all jobs
sacct -j <JOB_ID> --format=JobID,State,ExitCode,Elapsed,MaxRSS

# Example: sacct -j 7545235
```

### Read job output
```bash
# Check first job's stdout
cat logs/nmt-<ARRAY_JOB_ID>_1.out

# Check for errors
cat logs/nmt-<ARRAY_JOB_ID>_1.err

# Watch output in real-time (once a job starts)
tail -f logs/nmt-<ARRAY_JOB_ID>_1.out

# Count completed results
find results/ -name "test_metrics.json" | wc -l

# Count by model
find results/nllb-1.3b -name "test_metrics.json" 2>/dev/null | wc -l
find results/mbart-fr -name "test_metrics.json" 2>/dev/null | wc -l
find results/mbart-rand -name "test_metrics.json" 2>/dev/null | wc -l
```

### Check for failed jobs
```bash
# Jobs that exited with non-zero status
sacct -j <JOB_ID> --format=JobID,State,ExitCode | grep -v "COMPLETED\|PENDING\|RUNNING"

# Look at error logs for failed jobs
grep -l "ERROR\|Traceback" logs/*.err
```

### Cancel jobs
```bash
# Cancel all your jobs
scancel -u $USER

# Cancel a specific array
scancel <JOB_ID>

# Cancel specific array tasks
scancel <JOB_ID>_[100-200]
```

## 12. tmux Survival Guide

tmux keeps your session alive when you disconnect from SSH.
**Always use tmux for long-running HPC operations.**

### Essential commands

| Action | Command |
|--------|---------|
| New session | `tmux new -s nmt` |
| Detach (leave running) | `Ctrl+b` then `d` |
| Reattach | `tmux attach -t nmt` |
| List sessions | `tmux ls` |
| Kill session | `tmux kill-session -t nmt` |
| New window | `Ctrl+b` then `c` |
| Switch windows | `Ctrl+b` then `n` (next) or `p` (prev) |
| Split horizontal | `Ctrl+b` then `"` |
| Split vertical | `Ctrl+b` then `%` |
| Switch panes | `Ctrl+b` then arrow keys |
| Scroll mode | `Ctrl+b` then `[` (use arrows/PgUp, `q` to exit) |

### Recommended workflow
```bash
# SSH in
ssh f006g5b@discovery.dartmouth.edu

# Start or reattach tmux
tmux attach -t nmt 2>/dev/null || tmux new -s nmt

# Do your work...
sbatch --array=1-699 submit_array.sbatch
squeue -u $USER

# Detach when done monitoring
# Ctrl+b then d

# Later, reconnect
ssh f006g5b@discovery.dartmouth.edu
tmux attach -t nmt
```

## 13. Troubleshooting

### "Invalid account or partition"
Your account doesn't have access to the requested partition.
```bash
# Check what you have access to
sacctmgr show assoc where user=$USER format=account,partition,qos
```
Ask your PI to add you to their lab account, or use a partition you have access to.

### Jobs stuck in PENDING
```bash
# Check why
squeue -u $USER -o "%i %T %r" | head -20
```
Common reasons:
- `Resources` — waiting for GPUs to free up (normal, just wait)
- `QOSMaxJobsPerUserLimit` — too many jobs running, others will start as current ones finish
- `AssocMaxJobsLimit` — account-level limit

### "Container not found"
```bash
ls -lh nmt-training.sif
# If missing, rebuild:
bash build_container.sh
```

### No output in logs
- Check `.err` file: `cat logs/nmt-<ID>_<TASK>.err`
- Container sets `PYTHONUNBUFFERED=1` but if stdout is still buffered, the training script
  also calls `sys.stdout.reconfigure(line_buffering=True)` as a backup

### Out of disk space
```bash
df -h .
du -sh models/* results/* *.sif | sort -rh
```
Move to lab storage or scratch (see [Storage Management](#4-storage-management)).

### GRES string doesn't match
If the GPU resource name changes (cluster upgrades, etc.):
```bash
# See what's actually available
sinfo -p gpuq -O gres:60,nodelist | head -10
```
Update line 6 of `submit_array.sbatch` to match.

### Job ran out of time (4 hours)
Increase the time limit in `submit_array.sbatch`:
```bash
#SBATCH --time=06:00:00   # 6 hours instead of 4
```
Or reduce batch sizes / learning rate steps in `hf_job_train_hpc.py`.
